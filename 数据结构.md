# 数据结构

## 一、哈希 (Hash)

哈希表 (Hash Table)，也叫散列表，是很多高级语言（如 Python 的字典 `dict`，Java 的 `HashMap`）内部实现的核心。

### 核心思想

**一句话概括：通过一个函数，将“键”(Key)直接映射到一个内存地址，从而实现“即时”访问。**

想象一下查字典，你根据拼音（Key）直接翻到对应的那一页（内存地址），而不是从第一页挨个往后找。

### 重点内容回顾

1. **键值对 (Key-Value Pair):** 哈希表存储的基本单位。通过 Key 来存取 Value。
2. **哈希函数 (Hash Function):** 核心中的核心。它的作用是接收一个任意长度的 Key，输出一个固定长度的、整数类型的“哈希值”(Hash Code)。这个哈希值通常用来计算数组的索引。
   - 一个好的哈希函数应该具备：**确定性**（相同的输入总有相同的输出）、**高效性**（计算速度快）、**均匀性**（输出的哈希值应尽可能均匀分布，减少冲突）。
3. **哈希冲突 (Hash Collision):** 这是哈希表最关键的难点。当两个不同的 Key 经过哈希函数计算后，得到了相同的哈希值，就发生了冲突。**如何解决冲突**是哈希表学习的重点。
4. **解决冲突的方法:**
   - **链地址法 (Chaining):** 这是最常用、最重要的方法。在冲突发生的位置，数据不直接存储在数组里，而是存储一个**链表**（或其他数据结构，如红黑树）。所有冲突的元素都挂在这个链表上。查询时，先找到对应的链表，再在链表里查找。
   - **开放地址法 (Open Addressing):** 如果发生冲突，就去寻找下一个可用的空位。
     - **线性探测:** 依次尝试 `index+1`, `index+2`, ... 直到找到空位。缺点是容易产生“聚集”现象。
     - **二次探测:** 依次尝试 `index+1²`, `index+2²`, ...
     - **双重哈希:** 用第二个哈希函数来计算步长。

## **java的`HashMap`** 

**主要特点详解**

#### 1. 高性能的存取

这是 `HashMap` 最核心的优势。

- **平均时间复杂度 O(1)**：对于 `put(key, value)` 和 `get(key)` 操作，只要哈希函数设计得好，能将元素均匀分布，那么无论 `HashMap` 中有多少数据，操作的时间都基本是固定的。
- **最坏时间复杂度 O(n)**：当发生严重的哈希冲突（所有元素的哈希值都一样）时，哈希表会退化成一个链表（或红黑树），此时查找就需要遍历这个长链表/树，时间复杂度会下降到 O(n) 或 O(log n)。

#### 2. 键值对（Key-Value）存储结构

- 它实现了 `Map` 接口，用于存储键值对。
- **键（Key）是唯一的**：如果你用一个已经存在的 `key` 去执行 `put` 操作，新的 `value` 会覆盖旧的 `value`。
- **值（Value）可以重复**。

#### 3. 允许 `null` 键和 `null` 值

- `HashMap` 最多只允许一个 `key`为 `null`。
- `value` 可以有多个为 `null`。
- （作为对比：`Hashtable` 不允许任何 `null` 键或 `null` 值）。

#### 4. 无序性（Unordered）

- `HashMap` 不保证元素的存储和迭代顺序。也就是说，你插入元素的顺序和遍历（iterate）出来的顺序很可能是不同的。
- 这个顺序在扩容（resize）后还可能会发生变化。
- 如果需要保证顺序，应该使用：
  - `LinkedHashMap`：保证插入顺序或访问顺序。
  - `TreeMap`：保证键的自然排序或自定义排序。

#### 5. 非线程安全（Not Thread-Safe）

- `HashMap` 没有内置的同步机制。如果在多线程环境下对同一个 `HashMap` 实例进行修改操作（如 `put`），可能会导致数据不一致，甚至在扩容时产生死循环（在 JDK 7 中尤为突出）。
- 在多线程场景下，应该使用：
  - `Collections.synchronizedMap(new HashMap<>())`：通过包装器实现同步，但性能较差，因为所有操作都锁定了整个对象。
  - `ConcurrentHashMap`：推荐使用。它采用了分段锁（JDK 7）或 CAS + Synchronized（JDK 8+）等更优化的并发策略，提供了更高的并发性能。

## 内部实现原理（深入理解特点的根源）

`HashMap` 的内部结构是其高性能和特点的根源。

#### 1. 数据结构：数组 + 链表 / 红黑树 (JDK 8+)

- `HashMap` 内部维护一个**`Node[]` 数组**（数据中也叫哈希桶，Hash Bucket）。
- **`put` 过程**：
  1. 计算 `key` 的 `hashCode()`。
  2. 通过一个扰动函数对哈希码进行再处理，然后用 `(n - 1) & hash`（`n` 是数组长度）计算出在数组中的索引位置。
  3. 如果该位置没有元素，直接存入。
  4. 如果该位置已经有元素（**哈希冲突**），则以**链表**的形式挂在后面。
- **JDK 8 的优化**：当同一个哈希桶中的链表长度超过**8**，并且整个 `HashMap` 的容量大于**64**时，这个链表会**转化成红黑树**。红黑树的查找时间复杂度为 O(log n)，能有效改善极端哈希冲突下的性能。

#### 2. 两个重要参数

`HashMap` 的性能受两个参数影响，可以在创建时指定：

- **初始容量（Initial Capacity）**：哈希桶数组的初始大小，默认为 **16**。
- **加载因子（Load Factor）**：一个介于 0 到 1 之间的浮点数，默认为 **0.75**。它决定了 `HashMap` 在其容量自动增加之前可以达到多满。
  - **计算公式**：当 `HashMap` 中的元素数量 > `当前容量 * 加载因子` 时，就会触发**扩容（resize）**。
  - **扩容**：`HashMap` 会创建一个新的、容量为**原容量两倍**的数组，并将所有旧元素重新计算哈希位置并迁移到新数组中。这是一个相对耗时的操作。

## **1.尾插法**

①初始状态：

```bash
旧 table (容量 = 8)
+---+
| 0 |
+---+
| 1 |
+---+
|...|
+---+
| 5 | --▶ [ A ] -> [ B ] -> [ C ] -> [ D ] -> null
```

②扩容开始：

JDK 8 的 `resize` 开始了。它不是直接移动节点，而是创建了两条“虚拟”的链表，我们叫它 `lo` 链和 `hi` 链。

```bash
     原始链表: [ A ] -> [ B ] -> [ C ] -> [ D ] -> null
         ▲
         |
      (当前处理节点 e)

     lo 链 (位置不变):  null
     hi 链 (位置改变):  null
```



③开始分配：

```bash
1.
     原始链表: [ A ] -> [ B ] -> [ C ] -> [ D ] -> null
                 ▲
     
     lo 链: [ A ] -> null   (A成为lo链的头和尾)
     hi 链: null
     
     
2.
     原始链表: [ A ] -> [ B ] -> [ C ] -> [ D ] -> null
                         ▲
     
     lo 链: [ A ] -> null
     hi 链: [ B ] -> null   (B成为hi链的头和尾)
     
     
3.
     原始链表: [ A ] -> [ B ] -> [ C ] -> [ D ] -> null
                                 ▲
     
     lo 链: [ A ] -> [ C ] -> null   (C接在A的后面)
     hi 链: [ B ] -> null


4.
     原始链表: [ A ] -> [ B ] -> [ C ] -> [ D ] -> null
                                         ▲
     
     lo 链: [ A ] -> [ C ] -> null
     hi 链: [ B ] -> [ D ] -> null   (D接在B的后面)
```

④将新链表挂到新 `table`：

```bash
新 table (容量 = 16)

+---+
|...|
+---+
```

## **2.提升效率**

#### 第一部分：理解背景知识

要明白这个运算，我们先要回顾两个基本事实：

1. **HashMap 的容量永远是 2 的幂次方**：比如 16, 32, 64...
2. **计算索引的方法**：`index = hash & (capacity - 1)`。

我们用 `capacity = 16` 举例：

- `capacity` = 16 (二进制: `10000`)
- `capacity - 1` = 15 (二进制: `01111`)
- `hash & (capacity - 1)` 就是 `hash & 01111`。这相当于取 `hash` 值的低 4 位作为索引，因为 `&` 运算中，任何数与 `0` 相与都得 `0`，与 `1` 相与都保留原样。

------

#### 第二部分：扩容时的变化

现在，假设我们要从 `oldCapacity = 16` 扩容到 `newCapacity = 32`。

看看我们的“掩码”发生了什么变化：

- **旧掩码**: `oldCapacity - 1` = 15 = `01111` (二进制)
- **新掩码**: `newCapacity - 1` = 31 = `11111` (二进制)

**发现了吗？** 新旧掩码的唯一区别，就是多出来一个**高位的 `1`**。

这个多出来的高位 `1`，它的位置在哪里？
 `11111 - 01111 = 10000` (二进制)
 这个 `10000` 正好就是 `oldCapacity` (16)！

**这个新多出来的 bit 位，我们称之为“differentiating bit”（区分位）。**

------



#### 第三部分：`(e.hash & oldCapacity)` 的真正含义

现在我们来看这行代码。它实际上就是在检查一个节点的 `hash` 值，在**这个新多出来的“区分位”上，到底是 `0` 还是 `1`**。

我们用一个具体的 `hash` 值来模拟：

- `oldCapacity` = 16 (`0001 0000`)
- 假设一个节点的 `hash` = 85 (`0101 0101`)

**1. 扩容前的索引计算：**

 `hash & (oldCapacity - 1)`
 `0101 0101 & 0000 1111`  (即 `& 15`)
 结果是 `0000 0101`，十进制为 `5`。
 所以，这个节点在旧 table 中的索引是 `5`。

**2. 现在，我们用 `(e.hash & oldCapacity)` 来判断它的新去向：**
 `e.hash & oldCapacity`
 `0101 0101 & 0001 0000` (即 `& 16`)
 结果是 `0001 0000`，不为 `0`。

**3. 这说明了什么？**
 这说明该 `hash` 值的“区分位”（从右数第5位）是 `1`。
 当它和**新掩码** `11111` 进行与运算时，这个 `1` 会被保留下来。

我们来计算一下它在新 table 中的索引：
 `hash & (newCapacity - 1)`
 `0101 0101 & 0001 1111` (即 `& 31`)
 结果是 `0001 0101`，十进制为 `21`。

`21` 是什么？ `21 = 5 + 16`，也就是 **`原索引 + 原容量`**！

------

#### 动画演示与总结

让我们把这个逻辑用图画出来：

假设一个 `hash` 值，二进制表示为 `... ? xxxx`

- `xxxx` 是低位，决定了它在**旧 table** 中的索引。
- `?` 是那个关键的“区分位”。

```markdown
   hash:      ... ? x x x x
&  oldCap-1:  ... 0 1 1 1 1
---------------------------
   oldIndex:  ... 0 x x x x  (只看低位)

   hash:      ... ? x x x x
&  newCap-1:  ... 1 1 1 1 1
---------------------------
   newIndex:  ... ? x x x x  (低位和区分位一起看)

```

现在，`(e.hash & oldCapacity)` 就是在看这个 `?` 位是 `0` 还是 `1`。

- **情况 A: `(e.hash & oldCapacity) == 0`**
  - 这说明 `?` 位是 `0`。
  - `newIndex` = `...0xxxx` = `oldIndex`。
  - **结论：节点位置不变，留在原索引。**
- **情况 B: `(e.hash & oldCapacity) == 1`**
  - 这说明 `?` 位是 `1`。
  - `newIndex` = `...1xxxx`。
  - `...1xxxx` 和 `...0xxxx` 之间差了多少？正好差了那个 `1` 代表的权重，也就是 `oldCapacity`。
  - 所以 `newIndex` = `oldIndex + oldCapacity`。
  - **结论：节点位置改变，移动到 `原索引 + 原容量`。**

------



## JDK8相对JDK7优化了索引的平均分配问题

计算方法

```
索引  =  （链表长度-1） &  节点hash值
```

这也就说明，只有节点的hash值的最后几位会影响计算的结果，如果我们的数据的hash值低位都相似，就会导致大量hash冲突。

- jdk7是通过一个扰乱函数解决的。
- jdk8通过将高位和地位进行**异或**，得到一个新的值，然后拿新的值去&（链表长度-1）。



# 二.树 (Tree)：核心知识体系

#### 1. **基本术语**：

**节点**(Node)、**根**(Root)、**父**(Parent)、**子**(Child)、**叶**(Leaf)、**深度**(Depth)、**高度**(Height)。这是学习任何树都必须掌握的通用语言。



#### 2. 最简单和重要的模型：二叉树 (Binary Tree)

- **定义**：每个节点最多有两个子节点（左子节点、右子节点）。它是绝大多数复杂树结构的基础和研究起点。
- **关键类型**：
  - **满二叉树**：一个理想化的形态，除叶子节点外，所有节点都有两个子节点。
  - **完全二叉树**：更贴近现实的形态，除了最后一层，上面都是满的，且最后一层节点从左到右排列。**堆 (Heap)** 就是一个典型的完全二叉树应用。
- **核心操作**：**树的遍历**
  - **深度优先 (DFS)**：前序、中序、后序。这三种遍历是算法题的常客。
  - **广度优先 (BFS)**：层序遍历。通常用队列实现。

------

#### 3. 演进第一步：为了“高效查找” -> 二叉搜索树 (BST)

- **解决的问题**：普通的二叉树查找效率不高 (O(n))。如何让查找变快？
- **解决方案**：在树上增加一个**排序规则**。
- **核心性质**：对于任意节点，其**左子树所有节点的值 < 该节点的值 < 右子树所有节点的值**。
- **带来的优势**：利用这个性质，我们可以像“二分查找”一样搜索，使得插入、删除、查找的**平均时间复杂度**降到了 **O(log n)**。
- **致命缺陷**：在极端情况下（如插入一个有序序列），BST会**退化成一个链表**，所有操作的时间复杂度恶化为 **O(n)**。

------

#### 4. 演进第二步：为了“持续高效” -> 平衡二叉搜索树 (Balanced BST)

- **解决的问题**：如何防止BST退化，让它始终保持“平衡”，从而确保 O(log n) 的高效性能？
- **解决方案**：在每次插入或删除后，通过**“旋转”**等操作来自动调整树的结构，维持其平衡。
- **主流代表**：
  - **AVL树**：最严格的平衡树。要求任何节点的左右子树高度差不超过1。实现复杂，读操作极快，写操作调整频繁。
  - **红黑树 (Red-Black Tree)**：工程应用中最常见的平衡树。它通过一套相对宽松的规则（5条性质）来维持“大致的平衡”，性能非常稳定。**Java的 `TreeMap` 和 `HashMap` (链表过长时) 就用的它。**

> **至此，我们完美地解决了在【内存中】高效存储和查找有序数据的问题。**

------

#### 5. 演进第三步：为了适应“外部存储” -> B树 & B+树

- **面临的新问题**：当数据量巨大（上亿条），内存已经放不下，必须存储在**硬盘**上时，红黑树还高效吗？
  - **答案是不够高效**。因为硬盘I/O的速度远慢于内存，且硬盘是按“块”(Page)读取的。红黑树的高度相对较高，一次查询可能需要多次磁盘I/O，这会成为性能瓶颈。
- **核心思想**：**降低树的高度**，即使会增加每个节点内部的比较次数。因为CPU计算远快于磁盘I/O，所以要不惜一切代价减少I/O次数。
- **解决方案**：让树从“高瘦”变为“**矮胖**”。

#### 5.1 B树 (也叫B-树，B和B减是同一个东西！)

- **特点**：
  - **多路搜索**：每个节点可以有多个子节点（远超2个）。
  - **节点存数据**：每个节点都存储键(Key)和对应的数据(Data)。
  - **平衡**：所有叶子节点在同一层。
- **目标**：通过在单个节点内存储大量键，极大地降低树的高度，从而减少磁盘I/O次数。

#### 5.2 B+树 (B+ Tree) ，B树的终极优化版

- **解决的问题**：B树虽然已经很好，但能否更进一步？能否让范围查询更快？
- **核心优化**：
  1. **内部节点只存Key**：非叶子节点只作为索引，不存数据。这使得单个节点能容纳的Key更多，树的高度被进一步压缩，变得**更矮胖**。
  2. **数据全在叶子节点**：所有数据都只存在于叶子节点。这保证了任何查询的路径长度都相同，查询性能更稳定。
  3. **叶子节点形成链表**：所有叶子节点通过指针连接成一个有序的双向链表。
- **带来的王牌优势**：
  - **范围查询极其高效**：对于 `WHERE age > 20 AND age < 30` 这样的操作，只需在叶子层找到起点，然后顺着链表顺序遍历即可，无需回溯，I/O操作是连续的。
- **应用**：**现代数据库和文件系统的索引标准实现**。当你给MySQL的表字段加索引时，底层创建的就是B+树。

------

B+树基本只在数据库中应用，也在一些和数据库交互的底层开发平台应用，普通开发很少使用。

但是B树，尤其是B树的一个情况：2-3-4树，也就是红黑树，经常使用。

### 2-3-4树（红黑树）

2-3-4树：

【A，B，C】



红黑树：

```bash
     黑(B)
    /       \
  红(A)    红(C)
```

 注意点：红黑树（2-3-4树）在插入时有两种策略：

# ArrayList 和LinkedList

## ArrayList 

### 一、 核心概念：`ArrayList`是什么？

首先要一句话说清楚它是什么。

`ArrayList`是Java集合框架中的一个核心类，它**实现了`List`接口**，其底层是**基于动态数组**实现的。

你需要掌握它的几个关键特性：

1. **有序（Ordered）**: 元素存入的顺序和取出的顺序是一致的。
2. **可重复（Allows Duplicates）**: 可以存储重复的元素，`list.add("a"); list.add("a");` 是完全允许的。
3. **允许null值**: 可以存储`null`元素。
4. **基于索引（Index-based）**: 可以通过整数索引（下标）快速访问、修改元素，时间复杂度为 **O(1)**。
5. **非线程安全（Not Thread-Safe）**: 在多线程环境下，多个线程同时修改`ArrayList`可能会导致数据不一致或抛出`ConcurrentModificationException`异常。

------

### 二、 内部实现原理：面试必考！

这是理解其性能和行为的关键，也是面试官最喜欢问的地方。

#### 1. 底层数据结构

`ArrayList`内部维护了一个`Object[]`类型的数组 `elementData` 来存储元素。

```java
// transient: 表示这个字段不参与序列化
transient Object[] elementData; 
// size: 表示ArrayList中实际存储的元素个数，而不是数组的容量
private int size; 

```

**关键点**：

- `size` 和 `capacity` (数组长度) 是两个不同的概念。`size` 是你实际放进去的元素数量，`capacity` 是这个数组当前能容纳的最大元素数量。`size <= capacity`。

#### 2. 动态扩容机制（The `grow()` method）

这是`ArrayList`的精髓，也是最高频的面试题。

- **何时扩容？**：当你调用`add(E e)`方法时，`ArrayList`会检查当前`size + 1`是否会超过`elementData`数组的长度（`capacity`）。如果超过，就会触发扩容。
- **如何扩容？**：
  1. 计算新容量：新的容量通常是**旧容量的1.5倍** (`oldCapacity + (oldCapacity >> 1)`)。`>> 1`是右移一位，相当于除以2，效率比乘法`* 1.5`更高。
  2. 创建新数组：创建一个更大的新数组。
  3. 复制旧数据：使用`Arrays.copyOf()`方法将旧数组中的所有元素复制到新数组中。
  4. 指向新数组：让`elementData`引用指向这个新创建的、更大的数组。

**学习要点**：

- **扩容是昂贵的**：因为它涉及到数组的创建和数据复制。如果在创建`ArrayList`时能预估到数据量，最好使用带初始容量的构造函数 `new ArrayList<>(initialCapacity)`，这可以有效减少或避免扩容操作，提升性能。这是后端开发中一个常见的性能优化点。
- **首次添加元素**：当你用无参构造`new ArrayList()`创建实例时，内部数组`elementData`是一个空数组。第一次调用`add`方法时，会初始化一个默认容量为**10**的数组。

### 三、 核心操作及其时间复杂度

作为开发者，必须对常用操作的性能了如指掌。

| 操作             | 对应方法              | 时间复杂度      | 原因                                                         |
| ---------------- | --------------------- | --------------- | ------------------------------------------------------------ |
| **随机访问**     | `get(int index)`      | **O(1)**        | 基于数组索引，直接定位内存地址，非常快。                     |
| **末尾添加**     | `add(E e)`            | **O(1) (均摊)** | 大部分情况只需 `size++`。但若触发扩容，则是O(n)，不过这种情况不常发生，所以均摊下来是O(1)。 |
| **指定位置添加** | `add(int index, E e)` | **O(n)**        | 需要将 `index` 及其之后的所有元素向后移动一位，成本很高。    |
| **指定位置删除** | `remove(int index)`   | **O(n)**        | 需要将 `index` 之后的所有元素向前移动一位来填补空缺，成本很高。 |
| **按值删除**     | `remove(Object o)`    | **O(n)**        | 需要遍历数组找到该元素，然后执行移动操作。                   |
| **查找元素**     | `contains(Object o)`  | **O(n)**        | 需要遍历数组，逐个用`equals`方法比较。                       |

**后端开发启示**：

- `ArrayList`最适合的场景是：**大量的查询、遍历操作，以及在列表末尾的添加/删除操作**。
- 应**极力避免**在列表的开头或中间频繁地进行插入和删除，因为这会导致大量元素移动，性能极差。

### 四、 常见陷阱与最佳实践

这些是你在实际编码中会遇到的坑。

#### 1. 在循环中删除元素

错误的示例：

```java
// 错误方式1: for-i循环，会导致漏删
for (int i = 0; i < list.size(); i++) {
    if (condition) {
        list.remove(i); // 删除元素后，后面元素前移，i++会跳过下一个元素
    }
}

```

正确的做法是使用**迭代器（Iterator）\**进行删除，这是\**唯一安全**的方式。

```java
Iterator<String> iterator = list.iterator();
while (iterator.hasNext()) {
    String element = iterator.next();
    if (condition) {
        iterator.remove(); // 正确！这是迭代器的方法，不是list的方法
    }
}

```

#### 2. `Arrays.asList()` 的“坑”

`Arrays.asList()`返回的`List`**不是**`java.util.ArrayList`，而是`Arrays`的一个内部类。它**不支持**添加（add）和删除（remove）操作，因为它的底层仍然是原始数组，长度是固定的。

```java

List<String> list = Arrays.asList("a", "b", "c");
list.add("d"); // 会抛出 UnsupportedOperationException

```

如果需要一个可修改的`ArrayList`，正确的做法是：



```java

List<String> modifiableList = new ArrayList<>(Arrays.asList("a", "b", "c"));

```



#### 3. 线程安全问题

如前所述，`ArrayList`非线程安全。在并发场景下，如果你需要一个线程安全的`List`，有几种选择：

- `Vector`：远古时代的线程安全`List`，通过`synchronized`关键字实现，性能较差，基本**不推荐使用**。
- `Collections.synchronizedList(new ArrayList<>())`：一个包装器，将所有方法都用`synchronized`同步，性能一般。
- `CopyOnWriteArrayList`：**推荐在“读多写少”的并发场景下使用**。它在写入（add, set, remove）时会复制整个底层数组，在新数组上操作，操作完成后再将引用指向新数组。读操作不加锁，性能极高；写操作因为有复制成本，所以比较昂贵。

### 五、 `ArrayList` vs `LinkedList`

这是另一个经典的面试对比题，你需要从多个维度进行比较。

| 特性                       | `ArrayList`                  | `LinkedList`                              |
| -------------------------- | ---------------------------- | ----------------------------------------- |
| **底层结构**               | 动态数组                     | 双向链表                                  |
| **随机访问 (get)**         | **O(1)** - 极快              | O(n) - 慢，需从头/尾遍历                  |
| **插入/删除 (add/remove)** | **O(n)** (中间位置)          | **O(1)** (已知节点) / O(n) (需先查找)     |
| **内存占用**               | 较小，连续内存空间           | 较大，每个节点都有额外的prev/next指针开销 |
| **适合场景**               | **读多写少**，频繁的随机访问 | **写多读少**，频繁的插入和删除            |



### 六、扩容：

#### 第 1 步: `add(e)`

```java

// 这是我们最常用的添加元素的方法
public boolean add(e) {
    // 关键步骤：在添加元素前，确保容量充足
    ensureCapacityInternal(size + 1);  // 期望的最小容量是当前元素数量+1
    elementData[size++] = e; // 在数组末尾添加新元素，并增加size
    return true;
}

```

- **`size`**: `ArrayList` 中实际存储的元素数量。
- **`elementData.length`**: `ArrayList` 底层数组的实际长度，即**容量（Capacity）**。
- **`ensureCapacityInternal(size + 1)`**: 这是扩容机制的入口。它告诉 `ArrayList`：“我需要至少能容纳 `size + 1` 个元素的空间，请帮我检查并确保这一点。”

#### **第 2 步: `ensureCapacityInternal(int minCapacity)`**

这个方法是 `private` 的，作为内部调度的第一站。它的主要作用是**处理首次添加元素时的特殊情况**。

```java

private void ensureCapacityInternal(int minCapacity) {
    // 如果 elementData 是默认的空数组 (通过 new ArrayList() 创建时)
    if (elementData == DEFAULTCAPACITY_EMPTY_ELEMENTDATA) {
        // 就从 minCapacity 和 DEFAULT_CAPACITY(10) 中取一个较大值
        // 第一次 add 时, minCapacity 是 1, 所以最终会取 10
        minCapacity = Math.max(DEFAULT_CAPACITY, minCapacity);
    }
    // 将计算好的 minCapacity 传递给下一个帮助方法
    ensureCapacityHelper(minCapacity);
}

```

- **`DEFAULTCAPACITY_EMPTY_ELEMENTDATA`**: 这是一个静态的、共享的空对象数组。当我们使用无参构造函数 `new ArrayList()` 创建列表时，`elementData` 就指向它。这是一种**延迟初始化**的优化，避免了创建空列表时立即分配内存。
- **`DEFAULT_CAPACITY`**: 值为 **10**。
- **核心逻辑**: 如果 `ArrayList` 是通过 `new ArrayList()` 创建的，并且是第一次添加元素，那么它的初始容量将被确定为 **10**。否则，就直接将 `size + 1` 这个值传递下去。

#### **第 3 步: `ensureCapacityHelper(int minCapacity)`**

这是实际执行**“是否需要扩容”**判断的地方。

```java
private void ensureCapacityHelper(int minCapacity) {
    // 如果期望的最小容量 > 当前数组的实际长度 (容量)
    if (minCapacity - elementData.length > 0) {
        // 容量不足，调用 grow 方法进行扩容
        grow(minCapacity);
    }
    // 如果容量充足 (minCapacity <= elementData.length)，则什么也不做。
}
```

- **核心逻辑**: 这是一个简单的 `if` 判断。只有当期望的最小容量（`minCapacity`）真的超过了当前数组的长度时，才会启动真正的扩容流程 `grow()`。

#### **第 4 步: `grow(int minCapacity)` - 扩容的核心**

这是整个扩容机制的“心脏”，负责计算新容量、创建新数组和复制数据。

```java
private void grow(int minCapacity) {
    // 获取旧容量
    int oldCapacity = elementData.length;
    
    // 计算新容量：新容量 = 旧容量 + (旧容量 / 2)，即旧容量的 1.5 倍
    int newCapacity = oldCapacity + (oldCapacity >> 1);
    
    // 特殊情况1：如果按 1.5 倍扩容后，新容量仍然小于我们期望的最小容量
    // (这通常发生在调用 addAll 添加大量元素时)
    if (newCapacity - minCapacity < 0) {
        newCapacity = minCapacity; // 那么新容量就直接采用期望的最小容量
    }

    // 特殊情况2：防止新容量过大，超过了数组在JVM中能分配的最大值
    if (newCapacity - MAX_ARRAY_SIZE > 0) {
        newCapacity = hugeCapacity(minCapacity); // 处理极端情况
    }
    
    // 最关键的一步：创建一个新容量的数组，并将旧数组的元素全部复制过去
    elementData = Arrays.copyOf(elementData, newCapacity);
}

// hugeCapacity 的逻辑
private static int hugeCapacity(int minCapacity) {
    if (minCapacity < 0) // overflow
        throw new OutOfMemoryError();
    // 如果期望容量已经超过了最大限制，则返回 Integer.MAX_VALUE
    // 否则返回最大限制值
    return (minCapacity > MAX_ARRAY_SIZE) ?
        Integer.MAX_VALUE :
        MAX_ARRAY_SIZE;
}

```

**`MAX_ARRAY_SIZE`**: `Integer.MAX_VALUE - 8`。减去8是为了给某些JVM实现的对象头信息预留空间。

**核心逻辑**:

1. **计算目标容量**: 新容量默认为旧容量的 **1.5 倍** (`oldCapacity + (oldCapacity >> 1)` 使用位运算 `>> 1` 代替 `/ 2` 效率更高)。
2. **修正目标容量**: 如果 `addAll` 等操作导致 `minCapacity` 非常大，1.5倍扩容后仍然不够，则直接使用 `minCapacity`作为新容量。
3. **处理容量溢出**: 检查新容量是否超过了 `MAX_ARRAY_SIZE`，如果超过，则进行更严格的检查，防止 `OutOfMemoryError`。
4. **执行扩容**: 调用 `Arrays.copyOf()`，这是整个扩容操作中**最耗费性能**的一步。它会创建一个新数组并完成所有元素的拷贝。
5. **为何用copy**：之所以必须使用 `copy`（拷贝）这个方法，根本原因就在于Java中数组的内存存储形式是连续的。

## LinkedList

### **`LinkedList` 核心知识点总结**

#### 1. 底层数据结构

- **双向链表 (Doubly Linked List)**：每个元素（节点 `Node`）都包含三个部分：
  1. **数据本身 (`item`)**
  2. **指向前一个节点的引用 (`prev`)**
  3. **指向后一个节点的引用 (`next`)**
- **非循环**：头节点的 `prev` 指向 `null`，尾节点的 `next` 指向 `null`。
- **内存不连续**：节点对象在内存中是散乱分布的，通过引用连接。
- **核心指针**：`LinkedList` 类本身维护 `first` 和 `last` 两个指针，分别指向头节点和尾节点，这使得对头尾的操作极其高效。

#### 2. 核心性能特性（优缺点）

- **优点 (擅长什么？)**
  - **高效的插入和删除**：尤其是在**链表头部和尾部**，时间复杂度为 **O(1)**。在链表中间操作，虽然修改指针是 O(1)，但定位到该位置需要 O(n) 的时间。
  - **功能强大**：同时实现了 `List` 和 `Deque` (双端队列) 接口，可以灵活地用作 **列表、栈、队列**。
- **缺点 (不擅长什么？)**
  - **低效的随机访问**：`get(int index)` 操作的时间复杂度为 **O(n)**，因为它必须从头或尾开始遍历查找。
  - **更大的内存开销**：每个节点都需要额外的空间来存储 `prev` 和 `next` 两个引用，比 `ArrayList` 更耗费内存。

**核心结论**：在绝大多数后端“查询、遍历为主”的业务场景中，**优先选择 `ArrayList`**。仅在明确需要**频繁对集合首尾进行增删**或需要一个**队列/栈**实现时，才考虑使用 `LinkedList`。

#### 3. 核心API与用法

- **作为列表 (`List`)**: `add(e)`, `add(index, e)`, `get(index)`, `remove(index)`。**注意 `get` 和 `remove` 性能较差**。
- **作为队列 (`Queue` - FIFO)**:
  - **入队**: `offer(e)` / `offerLast(e)` (推荐，返回 `boolean`)
  - **出队**: `poll()` / `pollFirst()` (推荐，返回 `null` 若为空)
  - **查看队首**: `peek()` / `peekFirst()` (推荐，返回 `null` 若为空)
- **作为栈 (`Stack` - LIFO)**:
  - **入栈**: `push(e)` (等同于 `addFirst`)
  - **出栈**: `pop()` (等同于 `removeFirst`，为空时抛异常)

#### 4. 关键注意事项（必知陷阱）

1. **线程不安全**：`LinkedList` 和 `ArrayList` 都不是线程安全的。在多线程环境中，必须使用外部同步（如 `Collections.synchronizedList`）或改用 `java.util.concurrent` 包下的并发集合，如 **`ConcurrentLinkedQueue`** (高性能非阻塞队列) 或 **`LinkedBlockingDeque`** (阻塞队列)。
2. **禁止使用索引 for 循环遍历**：下面这种写法是**性能灾难**，时间复杂度为 O(n²)。

```java

// 错误示范!
for (int i = 0; i < list.size(); i++) {
    list.get(i); // 每次get都是O(n)
}

```

​	请用下面方法：

```java

//方法1（内部实现同方法2）
for (String s : list) { // 内部会使用迭代器，高效
    // ...
}


//方法2
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String s = it.next();
    // ...
}

```



#### 5. 后端应用场景（了解）

1. **轻量级消息/任务队列**：利用其队列特性实现先进先出。
2. **实现撤销/重做功能**：利用其栈特性。
3. **算法实现**：如广度优先搜索 (BFS) 中使用队列，某些需要链式操作的算法。
4. **Web爬虫**：管理待抓取的URL队列。

### **Java集合核心思想：面向接口编程备忘录**

#### **一、 核心原则：依赖抽象，而非具体**

这不仅仅是Java的技巧，更是优秀软件设计的基石。

- **口诀**：左边用接口，右边用实现。
- **正确示例**：`List<String> names = new ArrayList<>();`
- **错误示范**：`ArrayList<String> names = new ArrayList<>();` (应避免，除非特殊情况)

**为什么？**
 为了**解耦**！让你的代码不被某个具体实现“绑架”，未来可以轻松更换实现（如 `ArrayList` 换成 `LinkedList`）而无需修改大量业务代码。

### **二、 两步决策法：如何选择接口与实现**

忘掉死记硬背每个类，用“需求”来驱动你的选择。

#### **第1步：思考“我要做什么？”，选择最合适的【接口】**

问自己：我需要的数据组织方式是什么？

| 我的需求是...                               | 我应该选择的接口是... | 一句话场景         |
| :------------------------------------------ | :-------------------- | :----------------- |
| **按顺序存储，通过索引访问** (有序，可重复) | `List<E>`             | 电话本、播放列表   |
| **先进先出** (排队)                         | `Queue<E>`            | 银行排队叫号系统   |
| **后进先出** (叠盘子)                       | `Deque<E>` (用作栈)   | 浏览器“后退”按钮   |
| **存储唯一元素，不关心顺序** (去重)         | `Set<E>`              | 统计独立访客IP     |
| **存储键值对** (字典)                       | `Map<K, V>`           | 存储学生姓名和分数 |

#### **第2步：思考“怎么做得更好？”，选择最优的【实现类】**

问自己：基于我的主要操作，哪个实现性能最好？

| 如果我选择了...    | 我的主要操作是...                   | 我应该 `new` 的实现类是... | 关键原因                              |
| :----------------- | :---------------------------------- | :------------------------- | :------------------------------------ |
| `List`             | **频繁查询** (`get(i)`)，读多写少   | `ArrayList`                | **`RandomAccess`** (数组实现，查询快) |
|                    | **频繁增删** (尤其在头尾)，写多读少 | `LinkedList`               | **双向链表** (增删快)                 |
| `Queue` 或 `Deque` | 作为**队列**或**栈**使用            | `ArrayDeque`               | **性能通常最优** (数组实现的双端队列) |
|                    | 需要在队列/栈上进行增删操作         | `LinkedList`               | 功能全面，实现了 `Deque`              |
| `Set`              | **无顺序要求**，性能最高            | `HashSet`                  | **哈希表**实现，速度快                |
|                    | 需要**保持插入顺序**                | `LinkedHashSet`            | 哈希表 + 链表                         |
|                    | 需要**自动排序**                    | `TreeSet`                  | **红黑树**实现，元素需可比较          |
| `Map`              | **无顺序要求**，性能最高            | `HashMap`                  | **哈希表**实现，速度快                |
|                    | 需要**保持插入顺序**                | `LinkedHashMap`            | 哈希表 + 链表                         |
|                    | 需要**按Key自动排序**               | `TreeMap`                  | **红黑树**实现，Key需可比较           |

### **三、 特殊情况与警告**

1. **何时能用实现类引用？**
   - **唯一理由**：当你必须调用该实现类**特有**的、而接口中**没有定义**的方法时。
   - **典型案例**：`ArrayList` 的 `ensureCapacity()` 方法，用于性能优化。
2. **避坑指南**
   - **永远不要用 `Stack` 类**：它设计有缺陷。需要栈时，请使用 `Deque<E> stack = new ArrayDeque<>();`。
   - **永远不要用 `Vector` 类**：它是古老的、线程安全的 `ArrayList`，性能差。需要线程安全的List时，使用 `Collections.synchronizedList()` 或 `CopyOnWriteArrayList`。
   - **`subList()` 返回的是视图，不是拷贝**：修改子列表会影响原始列表，反之亦然。要小心使用。
   - **`PriorityQueue` 就是堆**：需要“堆”数据结构时，直接用 `PriorityQueue`。它默认是最小堆。
