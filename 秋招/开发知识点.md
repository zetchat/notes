# 一.[HTTP 请求与响应的结构]

HTTP 是一种基于客户端-服务器模型的无状态协议，主要通过请求和响应的方式进行通信。以下是 HTTP 请求和响应的详细结构。

HTTP 请求结构

HTTP 请求由四部分组成：请求行、请求头、空行和请求体。

#### 1.**请求行** 

请求行包含以下信息：

- **方法**：如 GET、POST、PUT、DELETE 等，表示客户端希望执行的操作。
- **请求 URI**：指定资源路径，例如 */index.html*。
- **HTTP 版本**：如 HTTP/1.1 或 HTTP/2。 示例：*GET /index.html HTTP/1.1*

#### 2.**请求头** 

请求头包含客户端的环境信息和请求的相关参数，例如：

- *Host*：目标服务器的主机名。
- *User-Agent*：客户端的浏览器和操作系统信息。
- *Accept*：客户端支持的 MIME 类型。
- *Content-Length*：请求体的大小（如果有）。

#### 3.**空行** 

用于分隔请求头和请求体。

#### 4.**请求体**

 请求体是可选的，通常用于 POST 或 PUT 请求，包含需要发送到服务器的数据，例如表单数据或 JSON。

HTTP 响应结构

HTTP 响应由四部分组成：状态行、响应头、空行和响应体。

#### 5.**状态行** 

状态行包含以下信息：

- **HTTP 版本**：与请求中的版本一致。
- **状态码**：三位数字表示请求的处理结果，例如： 200：成功 404：未找到资源 500：服务器内部错误
- **状态信息**：状态码的简短描述，例如 *OK*。 示例：*HTTP/1.1 200 OK*

#### 6.**响应头** 

响应头包含服务器的环境信息和响应的相关参数，例如：

- *Content-Type*：响应体的 MIME 类型，例如 *text/html*。
- *Content-Length*：响应体的大小。
- *Set-Cookie*：设置客户端的 Cookie。
- *Server*：服务器名称。

#### 7.**空行**

 用于分隔响应头和响应体。

#### 8.**响应体** 

响应体包含服务器返回的数据，例如 HTML 页面、图片或 JSON 数据。

示例

以下是一个典型的 HTTP 请求和响应示例：

**客户端请求**：

> GET /index.html HTTP/1.1
>
> Host: www.example.com
>
> User-Agent: Mozilla/5.0
>
> Accept: text/html
>
> Connection: keep-alive

**服务器响应**：

> HTTP/1.1 200 OK
>
> Date: Wed, 18 Apr 2024 12:00:00 GMT
>
> Server: Apache/2.4.1
>
> Content-Length: 12345
>
> Content-Type: text/html; charset=UTF-8
>
> <!DOCTYPE html>
>
> <html>
>
> <head>
>
> <title>Example Page</title>
>
> </head>
>
> <body>
>
> <h1>Hello, World!</h1>
>
> </body>
>
> </html>

# 二.完整的一个Spring MVC组件工作流程

#### 1. `DispatcherServlet` → `HandlerAdapter` (选择“执行官”)

- **场景**: `DispatcherServlet` 已经通过 `HandlerMapping` 找到了要执行的 `HandlerMethod`。但 `DispatcherServlet` 是个“甩手掌柜”，它自己不会执行。
- **问题**: “我手头有这么多 `HandlerAdapter`（处理 `@Controller` 的、处理 `HttpRequestHandler` 的、等等），到底哪一个**有能力**执行这个 `HandlerMethod`？”
- **解决**: `DispatcherServlet` 会遍历它所持有的 `HandlerAdapter` 列表，并对每个 `Adapter` 调用 `supports(handler)` 方法。`RequestMappingHandlerAdapter` 会说：“没错，这个 `@RequestMapping` 标记的方法是我的菜，我支持！”，于是 `DispatcherServlet` 就把任务交给了它。

#### 2. `HandlerAdapter` → `HandlerMethodArgumentResolver` (选择“参数专家”)

- **场景**: `RequestMappingHandlerAdapter` 接到任务，准备调用 Controller 方法。但方法有各种各样的参数（`@RequestParam`, `@RequestBody`, `Model`, `HttpServletRequest`...）。
- **问题**: “这个方法的第一个参数是 `@RequestBody User user`，我该怎么凭空造出一个 `User` 对象来？我的参数专家团队里，谁能搞定这个？”
- **解决**: `HandlerAdapter` 会遍历它所持有的 `HandlerMethodArgumentResolver` 列表，并对每个 `Resolver` 调用 `supportsParameter(parameter)` 方法。`RequestBodyArgumentResolver` 会说：“这个参数有 `@RequestBody` 注解，是我的活儿，我来解析！”，于是它被选中来处理这个参数。这个过程会对方法的**每一个参数**都执行一遍。

#### 3. `HandlerMethodArgumentResolver` → `HttpMessageConverter` (选择“格式转换工具”)

- **场景**: `RequestBodyArgumentResolver` 已经站出来说要处理这个参数了。它知道要去读 HTTP 请求体（Request Body）。
- **问题**: “请求的 `Content-Type` 是 `application/json`，我要怎么把这串 JSON 文本变成一个 `User` Java 对象？我的工具箱里有这么多转换器（JSON的、XML的、Protobuf的），该用哪个？”
- **解决**: `Resolver` 会遍历它所持有的 `HttpMessageConverter` 列表，并对每个 `Converter` 调用 `canRead(class, mediaType)` 方法。`MappingJackson2HttpMessageConverter` 会说：“我能读 `application/json`，也能把它转成 `User` 对象，交给我！”，于是它被选中，执行反序列化操作。

#### 4. `HandlerAdapter` → `HandlerMethodReturnValueHandler` (选择“善后专家”)

- **场景**: Controller 方法成功执行，返回了一个结果（比如一个 `User` 对象）。
- **问题**: “厨师（Controller）把菜做好了，我（`HandlerAdapter`）该怎么处理这个返回的 `User` 对象？是把它序列化成 JSON 返回给前端，还是把它当成模型数据去渲染一个页面？我的善后专家团队里，谁能处理这种情况？”
- **解决**: `HandlerAdapter` 会遍历它所持有的 `HandlerMethodReturnValueHandler` 列表，并对每个 `Handler` 调用 `supportsReturnType(returnType)` 方法。如果方法上标记了 `@ResponseBody`，那么 `RequestResponseBodyMethodProcessor` 就会说：“这个返回值需要序列化成响应体，这是我的专长！”，于是它被选中，调用合适的 `HttpMessageConverter`（同样经过筛选）将 `User` 对象序列化成 JSON 并写入响应。

# 三.JAVA反射

#### 1.**`Class` 类：反射的入口**

 这是所有反射操作的起点。要对一个类进行反射，首先就要获取它的 `Class` 对象。有三种常见方式：

```java
// 1. 通过类名.class
Class<String> clazz1 = String.class;

// 2. 通过对象.getClass()
String s = "Hello";
Class<? extends String> clazz2 = s.getClass();

// 3. 通过类的全限定名字符串 (最重要！Spring就是这么用的)
// Class.forName 会抛出 ClassNotFoundException 异常
Class<?> clazz3 = Class.forName("java.lang.String");

```

#### 2.**`Constructor` 类：代表类的构造方法**

 可以用来创建类的实例。

```java
// 获取构造方法
Constructor<String> constructor = clazz1.getConstructor(String.class);
// 通过构造方法创建实例
String newStr = constructor.newInstance("World"); // newStr 的值是 "World"
```

#### 3.**`Field` 类：代表类的字段（成员变量）**

 可以用来读写字段的值。

```java
// 假设我们有一个 Person 类
class Person {
    private String name = "张三";
}

Person p = new Person();
Class<?> personClass = p.getClass();
// 获取名为 "name" 的字段，即使是 private
Field nameField = personClass.getDeclaredField("name");

// 关键！暴力破解访问权限，否则无法访问 private 字段
nameField.setAccessible(true);

// 读取字段值
Object value = nameField.get(p); // value 是 "张三"

// 修改字段值
nameField.set(p, "李四"); // 现在 p 对象的 name 字段值变成了 "李四"
```

#### 4.**`Method` 类：代表类的方法**

 可以用来调用方法。

```java
// 假设 Person 类有一个 private 方法
// private void sayHello(String message) { ... }

Method sayHelloMethod = personClass.getDeclaredMethod("sayHello", String.class);
// 同样需要暴力破解
sayHelloMethod.setAccessible(true);
// 调用方法，第一个参数是调用该方法的对象，后面是方法的参数
sayHelloMethod.invoke(p, "你好！");
```

# 四.以我的项目为例，Spring如何使用反射实现IoC（控制反转）+DI（依赖注入）

#### 1.Spring 启动时的“蓝图绘制”阶段（组件扫描）

当 Spring 应用启动时，它会开始一个叫做**组件扫描（Component Scanning）**的过程。

- **发现蓝图**：Spring 会在你的项目包（例如 `com.sms.*`）里四处寻找带有特定注解（如 `@Component`, `@Service`, `@Repository`, `@RestController` 等）的类。

- **找到 `AdminServiceImpl.class`**：

  - Spring 发现了 `AdminServiceImpl` 类，看到它头上的 `@Service("adminServiceImpl")` 注解。
  - **Spring 的内心活动**：“哦！这是一个需要我管理的 Bean。它的名字叫 `adminServiceImpl`，它的类型是 `AdminServiceImpl`，并且它实现了 `AdminService` 接口。我得把这个‘蓝图’（Bean Definition）记下来。”
  - 于是，Spring 在它的内部“注册表”（一个 `Map` 结构）里记录下：
    - **Bean 名称**: `"adminServiceImpl"`
    - **Bean 类型**: `com.example.AdminServiceImpl` (全限定名)
    - **它实现的接口**: `AdminService.class`
    - **其他元数据**: 比如 `@Transactional` 注解，说明它需要事务管理（这是后话，属于 AOP 范畴）。

- **找到 `SystemController.class`**：

  - Spring 又发现了 `SystemController` 类，看到它头上的 `@RestController` 注解。

  - **Spring 的内心活动**：“好的，这又是一个需要我管理的 Bean。我来看看它的结构。”

  - **关键分析**：Spring 会通过**反射**来分析这个类。它发现 `SystemController` 有一个**唯一的构造函数**：

  - ```
    public SystemController(AdminService adminService, StudentService studentService, TeacherService teacherService)
    ```

    

  - Spring 会记录下这个构造函

  - 数以及它需要的所有参数类型：`AdminService.class`, `StudentService.class`, `TeacherService.class`。

  - 这个信息至关重要，Spring 现在知道了：“**要想创建 `SystemController`，我必须先准备好 `AdminService`、`StudentService` 和 `TeacherService` 这三个类型的 Bean。**”
  
  至此，扫描和蓝图绘制完成。Spring 容器里还没有任何真正的对象实例，只有一堆如何创建这些对象的“说明书”。

#### 2.IoC 和 DI 的“对象装配”阶段（Bean 实例化与注入）

现在，假设一个请求进来了，或者 Spring 默认需要初始化所有单例 Bean，它开始着手创建 `SystemController`。

**Step 1: 尝试创建 `SystemController`**

Spring 拿出 `SystemController` 的“说明书”，准备创建实例。但它发现，构造函数需要三个参数。它不能直接 `new SystemController()`。

**Step 2: 解决依赖 - 创建 `AdminServiceImpl` (IoC)**

- Spring 暂停创建 `SystemController`，转而去解决第一个依赖：`AdminService`。

- 它在自己的“注册表”里查找类型为 `AdminService` 的 Bean。它找到了名为 `"adminServiceImpl"` 的蓝图。

- **IoC 开始表演（使用反射创建对象）**：

  1.**获取 Class 对象**：Spring 内部执行类似代码：

  ```java
  // "com.example.AdminServiceImpl" 是从蓝图中获取的类全名
  Class<?> clazz = Class.forName("com.example.AdminServiceImpl");
  ```

  2.**获取构造函数**：`AdminServiceImpl` 没有显式定义构造函数，所以它有一个默认的无参构造函数。Spring 通过反射获取它：

  ```java
  Constructor<?> constructor = clazz.getDeclaredConstructor(); // 获取无参构造器
  ```

  3.**创建实例**：Spring 调用 `newInstance()` 来创建对象，这等同于 `new AdminServiceImpl()`：

  ```java
  Object adminServiceImplInstance = constructor.newInstance();
  ```

  

  - **控制反转（IoC）** 在这里体现得淋漓尽致：创建 `AdminServiceImpl` 对象的**控制权**不在你的代码里，而在 Spring 框架手中。它通过反射完成了对象的创建。
  - 创建完成后，Spring 会将这个 `adminServiceImplInstance` 放入它的“单例池”（一个缓存 `Map`）中，供后续使用。

  **Step 3: 解决其他依赖**

  Spring 会用同样的流程，去创建 `StudentServiceImpl` 和 `TeacherServiceImpl` 的实例，并放入单例池。

  **Step 4: 回到起点 - 最终装配 `SystemController` (DI)**

  现在，`SystemController` 所需的所有依赖（`AdminService`, `StudentService`, `TeacherService` 的实例）都已经在 Spring 的单例池里准备就绪了。

  - **DI 开始表演（使用反射注入依赖）**：

    1. **获取 Class 对象**：`Class<?> controllerClass = Class.forName("com.example.SystemController");`

    2. **获取带参数的构造函数**：Spring 在第一幕已经知道了要找哪个构造函数。它通过反射精确地找到这个构造函数：

       ```java
       Constructor<?> constructor = controllerClass.getConstructor(
           AdminService.class,
           StudentService.class,
           TeacherService.class
       );
       ```

       

    3. **从容器中获取依赖实例**：Spring 从它的单例池中，根据类型取出之前创建好的 `adminServiceImplInstance`, `studentServiceImplInstance`, `teacherServiceImplInstance`。

       **调用构造函数并注入依赖**：这是最关键的一步。Spring 调用构造函数的 `newInstance()` 方法，并将刚刚取出的依赖实例作为参数传进去：

       ```java
       Object systemControllerInstance = constructor.newInstance(
           adminServiceImplInstance,
           studentServiceImplInstance,
           teacherServiceImplInstance
       );
       ```

       

    4. **依赖注入（DI）** 在这里完美体现：`SystemController` 并没有主动去“要”依赖，而是 Spring 容器在创建它的时候，主动将它所需要的依赖**注入**（通过构造函数参数）给了它。


# 五.Spring注释

#### 1.Spring 核心注解（用于 Bean 管理和依赖注入）

这些注解是属于 Spring 的**核心容器模块**，主要负责 **IoC（控制反转）** 和 **依赖注入**。

- `@Component`：所有注解的基础，表示这个类是一个组件，交给 Spring 管理。
- `@Service`：业务逻辑组件（== `@Component` + 语义更清晰）。
- `@Repository`：数据访问组件（支持异常转换）。
- `@Autowired`：自动注入依赖。
- `@Qualifier`：指定注入哪个 Bean。
- `@Value`：注入配置文件中的值。

------

####  2.Spring MVC 注解（用于 Web 请求处理）

这些注解是 Spring MVC 提供的，用于处理 HTTP 请求、参数绑定、返回响应等功能。

- `@Controller` / `@RestController`
- `@RequestMapping` / `@GetMapping` / `@PostMapping` 等
- `@RequestParam` / `@RequestBody` / `@PathVariable`
- `@ResponseBody` / `@ModelAttribute`
- `@ExceptionHandler`（处理控制器异常）
- `@ControllerAdvice`（全局异常处理或数据绑定）



# 六.C和JAVA的引用区别

## C

在 C 和 C++ 中，一个 `.c` (或 `.cpp`) 文件被称为一个**“翻译单元”**或者**“源文件”**。它本身不是一个独立的程序，而是构成一个程序的一部分。

如果你的一个 `.c` 文件要用到另一个 `.c` 文件里定义的函数或变量，你通常会先为那个 `.c` 文件写一个 **`.h` (头文件)**，里面声明了它提供的函数和变量。

然后，你在这个要引用的 `.c` 文件里，使用 **`#include "xxx.h"`** 来**文本性地包含**那个头文件的内容。这就像把另一个文件的内容直接复制粘贴过来一样。最终，所有编译好的 `.o` (目标文件) 会被**链接器**连接起来，形成一个完整的可执行程序。

## JAVA

**一个 `.java` 文件通常只包含一个 `public` 类！**

- 是的，你的观察得没错！在 Java 中，一个 `.java` 源文件，**通常只包含一个 `public` 类型的类（或者接口、枚举）**，并且这个 `public` 类的名字**必须和文件名完全一致**！
- 例如，有个 `UserService.java` 文件，里面就应该定义一个 `public class UserService { ... }`。
- 当然，一个 `.java` 文件可以包含多个**非 `public` 的类**，但这些类只能在同一个文件中被其他类使用，不能被其他文件直接引用。这种情况下，文件名还是由那个 `public` 类决定。

**`import` 语句：Java 引用其他类的唯一方式！** 

​	Java **大量使用 `import`** 来引用其他文件（确切地说是其他包中的类）！

假设你有两个类：

- `com.example.myapp.UserService.java`
- `com.example.myapp.controller.UserController.java`

如果 `UserController` 要使用 `UserService`，它的代码会是这样的：

​	package com.example.myapp.controller; 
``` 声明这个类所在的包

import com.example.myapp.UserService; // 看清楚了！这就是 import 语句！
```

**重点：**

如果两个类在**同一个包（Package）** 里，它们之间互相引用是**不需要 `import` 语句**的！



# 七.classpath是JVM的底层核心

`classpath` 是 Java 语言设计之初就有的概念，它是 **JVM（Java 虚拟机）** 用来寻找和加载 `.class` 文件（编译后的 Java 字节码）和各种资源文件（比如 `.properties`、`.xml`、图片、文本文件等）的一组路径！

- **JVM 启动时就需要 `classpath`：** 当你执行 `java` 命令来运行一个 Java 程序时，你通常会指定 `classpath`。
  - 例如：`java -classpath myapp.jar com.example.MyMainClass` 或者 `java -cp . MyMainClass`
  - `java` 命令会告诉 JVM，去 `classpath` 指定的这些目录或 JAR 包里面找 `MyMainClass` 的 `.class` 文件，以及 `MyMainClass` 运行时需要的所有其他类和资源文件。
- **类加载器 (ClassLoader) 的工作基础：** JVM 内部的**类加载器 (ClassLoader)** 就是根据 `classpath` 来工作的！当你的程序需要用到某个类时，类加载器就会按照 `classpath` 中定义的顺序，在这些路径下去搜索并加载对应的 `.class` 文件。
- **资源加载的基础：** 不仅仅是 `.class` 文件，Java 程序中的 `getResource()` 或 `getResourceAsStream()` 等方法，也是依赖 `classpath` 来查找和加载资源的。

**所以，`classpath` 是 JVM 能够找到并执行你的 Java 代码和所需资源的基本机制！** 没有 `classpath`，JVM 就是个瞎子，啥也找不到，你的程序根本就跑不起来！



#### 1.**Maven 和 Spring 在 `classpath` 里扮演什么角色？**

他们扮演的是**管家**和**利用者**的角色，而不是发明者！

**Maven/Gradle (构建工具) - `classpath` 的管家和组织者！** 

Maven (以及 Gradle) 这样的构建工具，极大地**简化了 `classpath` 的管理**！

1. **自动收集依赖：** 当你在 `pom.xml` 里声明了 `dependencies`（比如 `spring-boot-starter-web`、`mysql-connector-java`），Maven 会自动从中央仓库下载这些 JAR 包，并把它们放到你本地的 Maven 仓库里 (`~/.m2/repository/`)。
2. **构建时生成 `classpath`：** 在你执行 `mvn package` 或者在 IDE 里运行项目时，Maven 会：
   - 把自己写的 Java 代码编译后的 `.class` 文件（在 `target/classes/` 目录）。
   - 把 `src/main/resources/` 下的所有资源文件（也复制到 `target/classes/` 目录）。
   - 以及所有项目依赖的 JAR 包。
   - **全部组织起来，生成一个完整的 `classpath` 列表**，然后传递给 JVM！
3. **简化运行命令：** 如果没有 Maven，你要运行一个带很多依赖的 Java 应用，你的 `java -classpath` 命令可能会长得吓人，需要手动把每个 JAR 包路径都列出来。Maven (或 Spring Boot Maven Plugin) 会帮你把这些事情都搞定，生成一个可执行的 JAR 包，里面已经包含了运行所需的 `classpath` 信息。

#### 2.**Spring / Spring Boot - `classpath` 的利用者！** 

Spring (以及 Spring Boot) 框架，是建立在 Java 平台之上的！它当然会**深度利用 `classpath` 机制**！

1. **配置文件加载：**
   -  `application.yml`、`logback-spring.xml` 这些配置文件，Spring Boot 都是默认从 `classpath` 中加载的！
   - 比如 `spring.config.location` 配置项，如果你指定 `classpath:/config/`，就是告诉 Spring 去 `classpath` 下的 `config` 目录找配置文件。
2. **组件扫描 (Component Scanning)：**
   - Spring 的 `@ComponentScan` 注解，就是告诉 Spring 去 `classpath` 下的指定包路径中，扫描并发现带有 `@Component`, `@Service`, `@Repository`, `@Controller` 等注解的类，然后把它们注册成 Spring Bean。
3. **Mapper/模板加载：**
   - 像 Mybatis 的 Mapper XML 文件 (`mapper-locations: classpath*:/mapper/**/*.xml`) 或 Thymeleaf/JSP 模板文件 (`spring.thymeleaf.prefix: classpath:/templates/`)，Spring Boot 也都是通过 `classpath` 机制来定位和加载的。

**结论：** Spring 和 Spring Boot 并没有创造 `classpath`，它们只是 Java 世界中的优秀应用框架，它们**广泛地依赖和使用了 JVM 的 `classpath` 机制**，来完成自动配置、组件查找、资源加载等核心功能。

#### 3.Spring Boot 项目中 `classpath` 包含哪些目录？

**你的项目编译输出目录 (Development / IDE)** 

- **对于 Maven 项目：**

  - `项目根目录/target/classes/`   这个目录是 Maven 在编译你的 Java 源代码 (`项目根目录/src/main/java`) 后，把编译好的 `.class` 文件存放的地方。

    

  - 同时， `项目根目录/src/main/resources/` 目录下的所有资源文件在编译时也会被复制到这个`项目根目录/target/classes/` 目录里！

- **对于 Gradle 项目：**

  - `项目根目录/build/classes/java/main/` (存放 `.class` 文件)
  - `项目根目录/build/resources/main/` (存放资源文件)
  - 类似 Maven，Gradle 也有它自己对应的编译输出目录。

**注意：** 当你在 IDE (如 IntelliJ IDEA) 中直接运行 Spring Boot 应用时，IDE 通常会配置它自己的一个输出目录，它会实时把你的修改编译同步到这个目录，然后把这个目录添加到 `classpath` 中。这个目录通常在项目根目录的 `.idea` 或 `build` 文件夹下的某个位置。

**项目依赖的 JAR 包 (Dependencies)** 

- 任何在 `pom.xml` (Maven) 或 `build.gradle` (Gradle) 中声明的依赖，比如 `spring-boot-starter-web`、`mysql-connector-java`、`mybatis-plus-boot-starter` 等等，它们对应的 JAR 包都会被添加到 `classpath` 中。
- 这些 JAR 包通常位于你本地 Maven 仓库的路径下 (例如 `~/.m2/repository/`)，或者 Gradle 缓存的路径下。当你的应用运行时，JVM 会找到并加载这些 JAR 包里的 `.class` 文件和资源文件。



# 八.实体类，Mapper，接口的实现的具体区别是啥？



实体类，只包含数据库的各个属性。

**`AdminMapper` (数据访问层/DAO层)：** **数据操作员**。它只负责和**数据库**直接打交道，执行最基础的 **CRUD（增删改查）*操作。它的“逻辑”就是*把 Java 对象变成 SQL 语句，或者把 SQL 结果变成 Java 对象**。

**`AdminService` (业务逻辑层/Service层)：** **业务经理**。它负责处理**业务规则、业务流程**。它会调用 `Mapper` 来获取数据，然后根据业务需求对数据进行加工、验证，甚至协调多个 `Mapper` 的操作，来完成一个**完整的业务功能**。

也就是说

#### 1.实体类 (Entity) ≈ C语言的结构体 

- **`Admin.java`** 这个实体类，它的核心价值就是**组织和承载数据**。
- C语言的 `struct Admin { int id; char name[50]; ... };` 也是完全一样的目的：将一堆相关的变量打包在一起，形成一个有意义的数据结构。

#### 2.Mapper ≈ 一些底层函数

- `AdminMapper` 接口定义了一组**最基础、最原子化**的数据库操作。每一个方法都只做一件最简单的事，比如 “增”、“删”、“改”、“查”。

#### 3.接口≈复杂实现

-  调用**mapper**的底层函数，实现复杂工作。

# 十.Mybatis-Plus



#### 如何判断你要实现的方法是要写在mapper层还是service层？

基础：Mybatis-Plus帮我们在mapper层写了一个BaseMapper，在service层写了接口IService以及它的实现ServiceImpl。
BaseMapper帮我们实现了基础的很多增删改查

ServiceImpl通过调用BaseMapper的方法进一步实现了复杂一点的各种方法。

#### 什么时候在Service层什么时候在Mapper层？

- ServiceImpl自带的方法都是基于单表操作的。如果涉及联表，聚合（HAVING）统计或者复杂的查询，我们要回到Mapper层（比如我们自定义的AdimMapper）在里面编写SQL，再回到Service层调用。
- 其他时候一般在Service完成即可。

#### 何时使用ServiceImpl方法，何时使用wapper？

- 我们在Service层（无论是Service层下面的controller还是serviceimpl）
- 一般对于静态的调用，直接使用ServiceImpl的方法，比如getId()save()
- 对于动态的，比如涉及if，while循环的逻辑，用warpper实现



#### 何时使用BaseMapper何时使用ServiceImpl的方法？

表面看，BaseMapper的方法ServiceImpl全都实现了，但是ServiceImpl通常我们希望它是面对事物的。

比如我们有一个方法：发布视频

**核心业务**：保存视频到数据库（必须完成）

**非核心，可以失败**：

1. 生成相关日志
2. 匹配相应的推荐算法权重

这个时候我们要单独用一个带@Transactional注释的方法实现核心业务，内部使用ServiceImpl方法。
另外两个用BaseMapper实现：

```java
@Service
public class ArticleServiceImpl extends ServiceImpl<ArticleMapper, Article> implements IArticleService {

    // 直接注入底层的Mapper，而不是整个Service
    @Autowired
    private LogMapper logMapper; 
    @Autowired
    private ConfigMapper configMapper;

    // 注意：事务注解只在需要它的方法上
    @Override
    public void publishArticle(Article article) {
        // 1. 调用一个独立的、带事务的私有方法来处理核心业务
        saveArticleInTransaction(article);

        // 2. 核心业务成功后，在事务外部执行非核心操作
        try {
            // 直接调用mapper，意图明确：就是一个简单的插入，不关心事务
            Log operationLog = new Log("发布文章: " + article.getTitle());
            logMapper.insert(operationLog); 
        } catch (Exception e) {
            // 日志记录失败，只打印错误，不影响主流程
            log.error("记录日志失败", e);
        }

        // 3. 直接调用mapper查询，意图明确：就是一次单纯的DB读取，不进入任何业务服务层
        Config config = configMapper.selectOne(new QueryWrapper<Config>().eq("key", "recommend_weight"));
        
        // ...后续异步操作
    }

    @Transactional // 事务边界精准控制在核心逻辑上
    protected void saveArticleInTransaction(Article article) {
        // this.save() 等价于 this.baseMapper.insert()
        // 在这里，使用 this.save() 是可以的，因为它在事务方法内部。
        this.save(article); 
        if (someCondition) {
            throw new RuntimeException("模拟核心业务失败"); // 失败会回滚
        }
    }
}

```
