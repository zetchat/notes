### Java基础核心突击目录 (目标：5-7天搞定)

#### 模块一：集合框架 (Collections) - 面试的绝对主角

**重点中的重点：`HashMap`** (请投入你70%的精力在这里)

1. **整体认知：**
   - **是什么？** 一个基于哈希表实现的、用于存储键值对(Key-Value)的集合。
   - **特点？** 非线程安全，允许Key和Value为null，存取速度快（平均O(1)）。
2. **底层数据结构（面试必考，必须画得出图）**
   - **JDK 1.7：** 数组 + 链表。
     - **重点：** 搞清楚一个元素放进去（put）的全过程。计算`hash` -> 定位数组下标（桶）-> 发现冲突 -> 在链表头部插入（头插法）。
     - **面试追问点：** 为什么用头插法？（官方认为是新插入的元素更可能被访问）。这个头插法在多线程下会导致什么问题？（**环形链表，CPU 100%**，这是经典考题）。
   - **JDK 1.8：** 数组 + 链表 + 红黑树。
     - **重点：** 相比1.7有什么优化？
     - **[优化1] 链表转红黑树：** 什么时候转？（**链表长度 ≥ 8 且 数组容量 ≥ 64**）。为什么要转？（避免链表过长导致查询性能下降到O(n)，红黑树能保证O(logn)）。
     - **[优化2] 插入方式：** 从头插法改为了**尾插法**。为什么？（核心是为了配合红黑树的引入，并修复了1.7多线程下的环形链表问题）。
     - **[优化3] 哈希算法：** 优化了哈希值的计算，减少哈希冲突。
3. **核心方法源码（你需要能用自己的话讲出流程）**
   - `put(key, value)`：上面已经提到了，这是核心，必须滚瓜烂熟。
   - `get(key)`：计算`hash` -> 定位数组下标 -> 如果是链表就遍历查找，如果是红黑树就按树的方式查找。
   - `resize()` (扩容)：
     - **什么时候触发？** 当`size`（元素个数）超过 `capacity`（数组容量） * `loadFactor`（加载因子，默认0.75）时。
     - **怎么扩容？** 创建一个**2倍**于原容量的新数组，然后将旧数组中的所有元素**重新计算哈希**并放到新数组的对应位置上。
     - **面试追问点：** 为什么容量总是2的N次幂？（为了让 `(n - 1) & hash` 这个位运算能快速且均匀地计算出数组下标，替代了取模运算，效率更高）。
4. **产出要求：**
   - 能徒手画出1.7和1.8的`HashMap`结构图。
   - 能流畅地给面试官讲一遍`put`和`resize`的过程。
   - 能清晰地说出1.8相对于1.7的三大优化点。

**次重点：`ArrayList` vs `LinkedList`**

1. **`ArrayList`**
   - **底层结构：** 动态数组 (`Object[]`)。
   - **重点：** 增删查改的**时间复杂度**。
     - **查/改 (get/set)：** O(1)，因为是数组，通过下标直接定位。
     - **增/删 (add/remove)：** 平均O(n)，因为涉及后续元素的**批量移动**。在末尾添加是O(1)。
   - **扩容机制：** 添加元素时如果容量不足，会创建一个**原容量1.5倍**的新数组，并把旧数据复制过去。
2. **`LinkedList`**
   - **底层结构：** 双向链表。
   - **重点：** 增删查改的**时间复杂度**。
     - **查 (get)：** O(n)，需要从头或尾开始遍历。
     - **增/删 (add/remove)：** 如果是操作头尾节点，是O(1)；如果是操作中间节点，查找需要O(n)，但插入/删除动作本身是O(1)。
   - **应用场景：** 需要频繁在集合头尾进行增删操作的场景。
3. **产出要求：**
   - 能在一分钟内说清楚两者的底层结构、时间复杂度差异和适用场景。

#### 模块二：并发编程 (JUC) - 进阶必备，区分普通人与高手的关键

**1. `ConcurrentHashMap`**

- **是什么？** 线程安全的`HashMap`。
- **如何实现线程安全？（面试核心）**
  - **JDK 1.7：分段锁 (Segment)**
    - **结构：** `Segment[]` 数组 + `HashEntry[]` 数组。`Segment`继承自`ReentrantLock`，你可以把它理解为一个“带锁的小`HashMap`”。
    - **原理：** 将整个哈希表分成N个`Segment`（默认16），每个`Segment`管理一部分哈希桶。当一个线程需要操作某个数据时，只需锁定其所在的`Segment`即可，其他线程仍然可以访问别的`Segment`。
    - **优点：** 相比于`Hashtable`（锁整个表），大大提高了并发度。
  - **JDK 1.8：`CAS` + `synchronized`**
    - **结构：** 和`HashMap`在1.8的结构一样（数组+链表+红黑树）。
    - **原理：** 放弃了分段锁，粒度更细。
      - 在**写入**（put）时，如果哈希桶是空的，直接使用**CAS**（Compare-And-Swap）尝试写入。
      - 如果哈希桶不为空（发生冲突），则使用 `synchronized` **锁住这个哈希桶的头节点**。锁的粒度从一个“段”缩小到了一个“桶”，并发性能更高。
- **产出要求：**
  - 能清晰地讲出1.7和1.8在实现线程安全上的原理和区别（分段锁 vs CAS+synchronized）。
  - 知道为什么1.8性能更好（锁的粒度更小）。

**2. `volatile` 关键字**

- **是什么？** Java虚拟机提供的最轻量级的同步机制。
- **两大作用（必须背下来）：**
  - **1. 保证可见性：** 当一个线程修改了被`volatile`修饰的共享变量的值，这个修改会立刻被刷新回主内存，并且其他线程在读取该变量时，会强制从主内存读取最新值，而不是用自己工作内存的缓存。
  - **2. 禁止指令重排序：** 通过内存屏障，确保`volatile`修饰变量前后的代码不会被编译器和处理器为了优化而随意打乱执行顺序。
- **面试追问点：** `volatile`能保证原子性吗？（**不能！** `i++`这种复合操作它无能为力）。它和`synchronized`有什么区别？（`volatile`是轻量级，只能修饰变量，不保证原子性，不会造成阻塞。`synchronized`是重量级，可修饰方法/代码块，保证原子性和可见性，会造成阻塞）。

**3. `synchronized` 关键字**

- **是什么？** Java提供的内置锁，一种悲观锁。
- **用法：** 修饰实例方法、静态方法、代码块。
- **原理（非常重要）：**
  - 每个对象都有一个关联的监视器锁 (monitor)。
  - 当线程进入`synchronized`代码块时，会执行`monitorenter`指令，尝试获取对象的monitor的所有权。
  - 退出时执行`monitorexit`指令，释放锁。
  - **锁升级（JDK 1.6 优化）：** 为了减少获取和释放锁的开销，引入了**偏向锁 -> 轻量级锁 -> 重量级锁**的升级过程。你需要大致了解每种锁的状态和升级时机。
    - **偏向锁：** 只有一个线程访问，锁会“偏向”这个线程。
    - **轻量级锁：** 有多个线程交替访问，但没有激烈竞争，通过CAS自旋尝试获取锁。
    - **重量级锁：** 竞争激烈，自旋失败的线程会被挂起，进入阻塞状态，等待被唤醒。这是最消耗性能的状态。

**4. 线程池 (ThreadPoolExecutor)**

- **为什么要用线程池？** （1）降低资源消耗（复用线程）；（2）提高响应速度（任务来了直接用）；（3）便于管理（统一分配、调优和监控）。
- **核心构造参数（七个，必须全部记住并理解）：**
  1. `corePoolSize`：核心线程数。
  2. `maximumPoolSize`：最大线程数。
  3. `keepAliveTime`：非核心线程的空闲存活时间。
  4. `unit`：存活时间的单位。
  5. `workQueue`：任务阻塞队列。
  6. `threadFactory`：线程工厂，用于创建线程。
  7. `handler`：拒绝策略（当队列满了，线程也达到最大时如何处理新任务）。
- **工作流程（面试必考）：**
  1. 新任务来了，判断核心线程数（`corePoolSize`）是否已满？没满，创建新核心线程执行任务。
  2. 如果核心线程满了，判断工作队列（`workQueue`）是否已满？没满，将任务放入队列。
  3. 如果队列也满了，判断总线程数是否达到最大值（`maximumPoolSize`）？没满，创建**非核心线程**执行任务。
  4. 如果总线程数也满了，触发**拒绝策略** (`RejectedExecutionHandler`)。
